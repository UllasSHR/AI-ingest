Reading your own code

You directed every line of this project. Now learn to read it. Tip: in the dark code panels below, lines marked ⌖ are clickable — tap one to see exactly what it does.

01The four questions

Whenever you look at code, ask these in order. They're the whole skill, compressed.

What does it do?Find where it starts (the entry point), then follow the data as it moves and changes.
Why is it here?Every line solves a problem. If you can't see the problem it solves, that's the line to ask about.
When will it break?Look at the inputs and assumptions. Empty list? Missing file? Network down? Each one is a crack.
How will I notice?Read the actual error. Check the real side-effect, not just "it ran." A 200 is not proof.

02How to read a file you've never read

  • Find the entry point. In your scripts it's the main() call at the bottom. Start there, not at line 1.
  • Follow the data. The whole project is one river: scraper.js → a JSON file → brain.js → a brief file → the web page.
  • Comments are the map. The grey italic lines tell you the why. Read those first.
  • Names are clues. fetchHN, makeItem, loadRecentlySeen — good names say what a thing does before you read how.

05"When will it break?" — the usual cracks

  • Missing / empty inputs. Empty array, missing file, null where an object was expected.
  • The network. Anything with fetch can fail, time out, or 404. Always check res.ok.
  • Untrusted text. LLM output, scraped HTML, user input — never assume it's well-formed.
  • Different environments. Laptop vs. CI vs. Vercel differ — files that exist locally, a read-only filesystem in production.
  • Time & state. "Today's file," "already seen," caches that go stale.
  • The literal terminal. Quotes, characters, exact branch names.

06How to notice it's broken

  • Read the error — don't flinch. The first line usually names the problem (ENOENT: no such file = a file is missing).
  • "It ran" is not "it worked." Check the real result: did the file get written? Does the page show the items? The most important habit.
  • Log the counts. Your scraper prints hn 95 items. If it said 0, you'd know instantly.
  • Bisect. Change one thing, re-check. Don't change five and guess.
  • Cap your retries. Same fix fails twice → change your theory, don't repeat it.
Predict it: npm run brief finishes with no error, but the site shows yesterday's stories. Where do you look first?
Check the real side-effect, not the exit code. "No error" only means it didn't crash. Open web/brief.json — is its date today's? If yes, the brief is fine and it's a page/cache issue. If no, the brief didn't write. Follow the data to the exact broken step.
