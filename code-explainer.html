01 · JAVASCRIPT BUILDING BLOCKS

One shape for many sources

from scraper.js — makeItem()

Adapter pattern

Hacker News, Reddit, RSS feeds and HuggingFace all return wildly different JSON. Instead of teaching the rest of the app about each one, every source funnels its data through makeItem and comes out in one identical shape. This is the adapter idea: normalize at the edge, so everything downstream is simple. 


      

      

        
Concept to keep

        

          Three small features doing a lot of work here:
          { title, url, ... } is destructuring — unpack named fields from
          an object argument. score ?? null is nullish coalescing — use
          score unless it's null/undefined.
          .slice(0, 2000) caps content so files don't bloat.
        

      

      

        Go deeper
        

          
    
            
  • Why ?? and not ||? || would also reject
              0 (a real score!) because 0 is falsy. ??
              only rejects null/undefined — exactly what you want for numbers.
    • 
            
  • (title || '').trim() guards against a missing title: fall back to
              an empty string so .trim() never crashes on undefined.
    • 
            
  • The lesson: define the contract once (this shape) and every new source
              is just "map your fields into makeItem." Adding a 6th source is a few lines.
    • 
          

        

      

      
Try this: add a new source function that returns makeItem(...) — the page will render it with zero other changes.

    


    
    

      
02 · JAVASCRIPT BUILDING BLOCKS

      
Reaching into deep objects without crashing

      
from brain.js — reading Gemini's response

      

        The model's reply is deeply nested, and any level could be missing on a bad response.
        Optional chaining (?.) walks the path and quietly yields
        undefined instead of throwing "cannot read property of undefined".
      

      

        
        

      

      

        
Concept to keep

        

          a?.b?.c stops at the first null/undefined and
          returns undefined — no crash. Then if (!text) throw turns
          that into one clear error you control instead of a cryptic stack trace deep
          in someone else's data.
        

      

      

        Go deeper
        

          
    
            
  • ?.[0] is the array form — safe index access. ?.() exists too, for calling a maybe-missing function.
    • 
            
  • Pattern: access defensively, then fail loudly on your terms. Don't let the program limp on with undefined silently flowing downstream — that's how you get the "200 OK but nothing saved" bugs.
    • 
          

        

      

    


    
    

      
03 · RESILIENCE PATTERNS

      
Run everything in parallel — without one failure killing the rest

      
from scraper.js — main()


      
      

        
Promise.allSettled

        
      


      

        Five network calls fire at once. If Reddit is down, you still want the other four.
        Promise.allSettled waits for all of them and never rejects — each
        result is tagged fulfilled or rejected so you decide what to do.
      

      

        
        

      

      

        
Concept to keep

        

          Promise.all vs Promise.allSettled: all rejects
          the instant any promise fails (all-or-nothing). allSettled always
          resolves with a status for each. For "best effort, collect what you can," reach for
          allSettled.
        

      

      

        Go deeper
        

          
    
            
  • all.push(...r.value) — the ... spread flattens one source's array of items into the combined list.
    • 
            
  • Each task is started before the await (the functions are called when the array is built), so they run concurrently, not one-after-another. The await just waits for the batch.
    • 
            
  • This is the difference between a scraper that takes ~2s (parallel) and one that takes ~10s (sequential).

04 · RESILIENCE PATTERNS

      
The fallback chain: keep trying until something works

      
from brain.js — callGemini()


      
      

        
Fallback chain

        
      


      

        Google retires model names on their own schedule. Rather than hard-coding one and
        breaking overnight, the code tries each model in order. A "bad model name" error
        (400/404) means skip to the next; any other error is real and should stop.
      

      

        
        

      

      

        
Concept to keep

        

          The key judgement is retryable vs. fatal. continue moves to the
          next option for errors you can recover from; throw stops for ones you
          can't. Remembering lastErr means that if everything fails, the
          final message tells you why the last attempt failed — not just "it broke."
        

      

      

        Go deeper
        

          
    
            
  • Same shape applies far beyond LLMs: API endpoints, payment providers, CDN regions — "try A, fall back to B, then C."
    • 
            
  • Distinguishing error kinds is what separates a smart retry from a dumb one. Retrying a 401 Unauthorized forever is pointless; retrying a flaky 503 is sensible.
    • 
            
  • This pattern is written straight into your CLAUDE.md as a known landmine — that's how a hard-won lesson becomes reusable.
    • 
          

        

      

    


    
    

      
05 · RESILIENCE PATTERNS

      
De-duplicate with a Set

      
from scraper.js — fetchHN()

      

        The same HN story turns up under several keyword searches (AI,
        LLM, Claude...). This keeps the first occurrence of each URL
        and drops the rest, in a single pass.
      

      

        
        

      

      

        
Concept to keep

        

          A Set answers "have I seen this already?" in roughly constant time.
          seen.has(x) then seen.add(x) inside a .filter()
          is the canonical de-dupe. The same idea scales to "have I shown this story this week?"
          — which is exactly what card 9 does.
        

      

      

        Go deeper
        

          
    
            
  • Why not array.includes()? That re-scans the whole array every time — O(n) per check, O(n²) overall. A Set is O(1) per check.
    • 
            
  • .filter() returns a new array; the original isn't mutated. Returning new data instead of editing in place keeps functions predictable.
    • 
          

        

      

    


    
    

      
06 · RESILIENCE PATTERNS

      
Pausing inside async code

      
from scraper.js — between HN queries

      

        HN's search API rate-limits rapid-fire requests. There's no built-in
        sleep() in JavaScript, so you wrap setTimeout in a promise
        and await it — a 250 ms breather between queries.
      

      

        
        

      

      

        
Concept to keep

        

          new Promise(r => setTimeout(r, ms)) is the idiomatic "sleep" in JS.
          The promise resolves when the timer fires, and await pauses
          this function without freezing the program. Memorize this one-liner.
        

      

      

        Go deeper
        

          
    
            
  • It doesn't block the thread — other timers, I/O, and callbacks keep running. JS is single-threaded but non-blocking; await yields rather than spins.
    • 
            
  • Being a polite API citizen (delays, a real User-Agent) is what keeps free endpoints from blocking you. Both appear in this scraper.
    • 
          

        

      

    


    
    

      
07 · WORKING WITH AN LLM

      
Parsing JSON that a model "mostly" returned

      
from brain.js — parseJSON()

      

        Ask a model for JSON and it often wraps it in ```json fences or adds a
        chatty sentence. JSON.parse on that raw text throws. This strips the
        fences and grabs everything between the first {/[ and the
        last }/] before parsing.
      

      

        
        

      

      

        
Concept to keep

        

          Never trust model output as well-formed. Treat it like untrusted input from the
          network: clean it, then parse defensively. This little function is the seam between
          "the model said something" and "my code has real data."
        

      

      

        Go deeper
        

          
    
            
  • The regexes peel off a leading ```json (or plain ```) and a trailing fence.
    • 
            
  • Finding the first bracket and last closer salvages JSON even when the model adds prose like "Here you go:" before it.
    • 
            
  • It still throws if there's truly no JSON — which is correct. The goal is tolerance of formatting noise, not silently swallowing a genuine failure.

08 · WORKING WITH AN LLM

      
The two-stage pipeline (the cost trick)

      
from brain.js — main()


      
      

        
Two-stage filter

        
      


      

        Sending the full text of ~100 articles to the model would be slow and could blow the
        free tier. So stage 1 sends only titles and asks for the best indices; stage 2
        sends the full content of just those survivors. Cheap filter, then expensive detail.
      

      

        
        

      

      

        
Concept to keep

        

          Do the cheap discriminating work first. Asking for indices (not the
          rewritten items) keeps stage 1's output tiny and lets you map back to your own trusted
          objects with chosen.map(i => items[i]). .filter(Boolean)
          drops any bad index the model invented.
        

      

      

        Go deeper
        

          
    
            
  • This is "coarse filter → fine rank," the same shape as a DB index narrowing rows before an expensive sort, or a cheap model gating an expensive one.
    • 
            
  • Returning indices instead of full objects also means the model can't quietly corrupt your data — you keep the originals and just use its ordering.
    • 
            
  • temperature: 0.4 keeps the picks fairly consistent run-to-run; the prompts inline profile.md so the filter is personalized.
    • 
          

        

      

    


    
    

      
09 · WORKING WITH AN LLM

      
Memory across days, with a sliding window

      
from brain.js — loadRecentlySeen() / prune


      
      

        
7-day sliding window

        
      


      

        A hot story trends for days. Without memory the brief would repeat it every morning.
        The fix: remember the URLs you've already shown, skip them for 7 days, and prune
        anything older so the memory file can't grow forever.
      

      

        
        

      

      

        
Concept to keep

        

          A tiny JSON file (seen.json) is enough to give a stateless script
          state between runs. The two-part discipline: load & filter on the
          way in, record & prune on the way out. The time-window cutoff is what
          keeps it bounded.
        

      

      

        Go deeper
        

          
    
            
  • Object.entries(seen) turns {url: date} into [url, date] pairs you can .filter() and .map().
    • 
            
  • Reading returns a Set for fast .has() checks in the main loop — the de-dupe pattern from card 5, now persisted.
    • 
            
  • The empty-day branch (when everything's been seen) writes an empty brief and skips the LLM call entirely — don't pay for work with no input.

10 · THE WEB APP

      
React: hydrate from storage, then derive everything

      
from web/app/brief-list.js — BriefList

      

        The page remembers what you've read in localStorage. Two ideas here:
        load that state after mount with useEffect (storage doesn't
        exist during server render), and derive the visible/read lists from state
        rather than storing them separately.
      

      

        
        

      

      

        
Concept to keep

        

          Derived state, not duplicated state. unread, visible
          and read are all computed from items +
          readUrls on every render. There's one source of truth; the rest follows.
          Duplicating them into their own useState is how lists drift out of sync.
        

      

      

        Go deeper
        

          
    
            
  • useEffect(... , [readKey]) runs after the component mounts in the browser, so touching localStorage is safe (it's undefined on the server).
    • 
            
  • Each handler updates React state and writes storage together, so a refresh restores exactly what you saw.
    • 
            
  • Showing only unread.slice(0, BATCH) means the next-best items slide in as you clear the top — finite and ranked, never an endless wall.
    • 
          

        

      

    


    
    

      
11 · THE WEB APP

      
Closing the loop: a rating teaches tomorrow's brief

      
from brief-list.js & web/app/api/rate/route.js


      
      

        
Feedback loop

        
      


      

        Click a star → the UI saves it to localStorage instantly and
        fire-and-forgets a POST to the server → the route upserts it into
        feedback.json → tomorrow brain.js reads those ratings
        into the prompt. The product literally learns from you.
      

      

        
        

      

      

        
        

      

      

        
Concept to keep

        

          Fire-and-forget: the UI doesn't wait for the server — localStorage
          is the source of truth for what you see, so the network write can fail without breaking
          the page (.catch(() => {})). On the server, read-modify-write with an
          upsert: find the existing record by URL, replace it if present, else push a new one.
        

      

      

        Go deeper
        

          
    
            
  • findIndex returning -1 is the "not found" signal — i >= 0 ? replace : push is the upsert.
    • 
            
  • The route guards its input (!entry?.url || typeof entry.rating !== "number") and wraps everything in try/catch — never trust the request body.
    • 
            
  • The honest caveat (in the file's own comment): on Vercel the filesystem is read-only, so deployed ratings live only in the browser. Knowing a pattern's limits is part of knowing the pattern.
    • 
          

        

      

      
The big picture: raw data → cheap filter → personalized brief → your ratings → back into the filter. That feedback edge is what turns a scraper into a product.
