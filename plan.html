AI-ingest

A personal filter for the AI firehose — built front to back, the way real developers do it.

The Problem

Twitter sells you urgency. Every day looks like "the biggest week in AI." Most of it is noise dressed as news. You want signal — only the things that change what you can do tomorrow.

The Tool, in One Line

Each morning, a single page tells you: here are the 3–5 AI things from the last 24h that actually matter to you, and how each one is useful in your hands.

How It Works

1. Collectscrape sources
2. Scorerank by your interests
3. SummarizeLLM with personal context
4. Serveyour morning page
runs once daily · you read for 3 minutes · close the tab

The Plan

01 Set up like a real developer ~1 evening

Before any feature code. The goal of this phase is purely the workflow — so every later phase teaches Git by repetition, not by reading docs.

  • Install git, sign in to GitHub from VS Code.
  • git init in this folder. First commit: just this plan.html.
  • Create the repo on GitHub (private), git push.
  • Write a README.md with the one-line pitch and the 4-step flow above.
  • Add a .gitignore for .env, node_modules, .next.
Git you'll learn here init · add · commit -m · push · .gitignore · what "staged" vs "tracked" actually means.
02 Define what "personally important" means — in writing ~1 hour

This is the most important file in the project. The whole tool is a function of this document. Without it, you've built another generic news aggregator.

  • Create profile.md: what you're building, what you use day-to-day (Claude Code, Next.js, Supabase, etc.), what you ignore (academic benchmarks, fundraising news, etc.).
  • Write 5 example tweets — 3 you'd want to see, 2 you'd want filtered. These become your test cases.
  • Commit this. It will change as you learn what you actually want.
Git you'll learn here Meaningful commit messages. Compare "update" vs "profile: drop fundraising news from interests" — future-you reads the second one and understands.
03 Get raw tweets in — the ugliest possible version ~1 evening

One script. Pulls a day of tweets from ~15 accounts you pick. Dumps to a JSON file. No website yet, no AI yet. Just prove you can get the data.

  • Pick collection method: official Twitter API (paid), twscrape, or RSS bridges like Nitter/RSSHub. Start with whatever has the least setup friction.
  • Hard-code 10–20 source accounts in a sources.json.
  • Output: data/2026-05-21.json. Eyeball it. Is the content there?
  • Verify by reading it back — don't trust that "the script ran". (Lesson from past projects: a 200 OK is not proof.)
Node.js or Python cron / Task Scheduler flat JSON files
Git you'll learn here branch + checkout -b feature/scraper. Build on a branch, merge when it works. This is how real teams keep main always-working.
04 Add the brain — LLM ranks and summarizes ~1–2 evenings

One more script. Reads yesterday's JSON, feeds it to Claude (or any model) along with profile.md, gets back the top 3–5 items with a "why this matters to you" paragraph each.

  • Prompt design is the whole game here. Spend time on it. Iterate against your test tweets from Phase 02.
  • Cluster near-duplicates before ranking — 30 tweets about the same launch should count as one item.
  • For each item: what changed, why it's relevant to you specifically, one concrete thing to try.
  • Output: data/2026-05-21.brief.json.
Anthropic SDK prompt caching structured output (JSON)
Git you'll learn here Open a Pull Request to merge your feature branch — even solo. Read your own diff before merging. This catches more bugs than you'd guess.
05 Build the morning page ~1 evening

Now the website. Deliberately boring: a list of cards, the date at the top, no infinite scroll, no like buttons, nothing to pull you back in. The opposite of Twitter.

  • Next.js, deployed to Vercel. Reads today's .brief.json, renders it.
  • Each card: headline, your "why it matters", source link, "mark as read".
  • Once you've read it, the page goes empty until tomorrow. That's the whole point.
Next.js Vercel Tailwind
Git you'll learn here Connect the GitHub repo to Vercel. Every git push auto-deploys. This is the moment GitHub stops feeling like homework and starts feeling like a superpower.
06 Automate — it runs without you ~1 evening

The tool is only useful if it's there before you open Twitter. So it has to run on a schedule, not on your laptop.

  • GitHub Actions: cron job at 6am, runs scrape → summarize → commit the new brief back to the repo.
  • Vercel redeploys on commit. Page is fresh when you wake up.
  • Optional: a one-line email or push notification with the headlines.
Git you'll learn here GitHub Actions — YAML workflows, secrets management (API keys never go in the repo), watching CI runs. This is the same tooling real production systems use.
07 Tune for a month ongoing

The first week's briefs will be wrong in interesting ways. Items that shouldn't have made it. Items it missed. This is where the tool becomes yours.

  • Add a thumbs-up / thumbs-down on each card. Log it.
  • Every Sunday, read your downvotes. Update profile.md. Commit with a message like "profile: filter out new-model-release hype unless cheaper than Sonnet".
  • After a month, your profile.md is the most accurate description of your own attention that has ever existed in writing.
Git you'll learn here git log as a journal. Six months from now you can read the history of profile.md and see how your interests evolved. This is one of the secretly best uses of version control.

What you'll know how to do, after

  • Stand up a real project from empty folder to deployed site.
  • Use Git the way working developers do — branches, PRs, meaningful history, CI.
  • Wire an LLM into a product with a clear job and a measurable output.
  • Tell the difference between "ships" and "works" — and verify the second.

Ground rules I'd suggest

  • One commit per working thing. If you can't describe what changed in a sentence, the commit is too big.
  • Never push secrets. .env.local stays local. Always.
  • Build the read-back before declaring done. If the scraper "wrote 30 tweets", read 30 tweets back out before moving on.
  • The profile doc is the product. Code is just the delivery mechanism.
